# 数据流转与指令生成系统一期设计

## 1. 背景与目标

本项目一期聚焦两个目标：

1. 打通数据在平台内的真实流转链路，包括注册审核、登录鉴权、目录发布、需求发起、需求审批、数据上传、处理任务创建、结果交付。
2. 将“指令生成”作为第一个数据处理能力接入平台，但一期只做半接入，不直接执行真实 LLM 生成逻辑。

一期的交付目标不是做成完整的大平台，而是做成一个可在实验室服务器部署、支持外网访问、可供内部真实试用的最小可用系统。

## 2. 一期范围

### 2.1 纳入一期的能力

- 真实用户注册、管理员审核、登录、角色权限控制
- 数据提供者发布和管理数据目录
- 数据汇聚者浏览目录并发起数据需求
- 数据提供者审批需求并上传原始数据文件
- 数据汇聚者创建清洗、切分、指令生成任务
- 数据使用者查看并下载最终交付结果
- 关键操作日志记录

### 2.2 不纳入一期的能力

- 拆书能力的真实接入与执行
- 修改病句能力的真实接入与执行
- 指令生成真实模型调度与异步执行
- 多组织、多租户
- 复杂工作流编排
- 对象存储、消息队列、微服务拆分

## 3. 角色设计

一期包含四类角色。

### 3.1 管理员

职责：

- 审核注册申请
- 分配和调整业务角色
- 启用、停用用户
- 查看全局目录、需求、任务、日志

限制：

- 不直接参与数据目录发布、需求发起、数据处理和交付消费

### 3.2 数据提供者

职责：

- 发布、编辑、归档自己的数据目录
- 查看针对自己目录发起的数据需求
- 审批或拒绝需求
- 在审批通过后上传原始数据
- 查看与自己相关的处理和交付进展

### 3.3 数据汇聚者

职责：

- 浏览已发布的数据目录
- 发起数据需求
- 在数据上传完成后创建处理任务
- 管理清洗、切分、指令生成任务
- 确认处理结果并发起交付

### 3.4 数据使用者

职责：

- 查看已交付的数据结果
- 下载可使用的交付文件

限制：

- 一期不参与需求发起、审批、处理和反馈回流

## 4. 核心业务流程

### 4.1 用户与审核流程

1. 用户提交注册申请。
2. 注册申请进入待审核状态。
3. 管理员审核通过后为用户分配角色。
4. 用户使用审核通过的账号登录系统。
5. 若申请被拒绝或账号被停用，则用户不可登录。

### 4.2 数据流转主链路

1. 数据提供者发布数据目录。
2. 数据汇聚者浏览目录并发起数据需求。
3. 数据提供者审批需求。
4. 审批通过后，数据提供者上传原始数据文件。
5. 数据汇聚者基于上传文件创建处理任务。
6. 处理任务按需要经过清洗、切分、指令生成。
7. 数据汇聚者确认处理结果后将需求标记为已交付。
8. 数据使用者查看并下载交付结果。

## 5. 状态机设计

### 5.1 用户状态

- `pending_review`：已注册，待管理员审核
- `active`：审核通过，可正常登录
- `rejected`：注册申请被拒绝，不可登录
- `disabled`：账号被管理员停用，不可登录

### 5.2 数据目录状态

- `draft`：草稿态，仅提供者可见和编辑
- `published`：已发布，汇聚者可浏览和发起需求
- `archived`：已归档，不能再发起新需求

### 5.3 数据需求状态

- `pending_approval`：待提供者审批
- `approved`：已审批通过，待上传原始数据
- `rejected`：已被拒绝
- `data_uploaded`：原始数据已上传
- `processing`：已创建处理任务并处理中
- `delivered`：结果已交付

### 5.4 处理任务状态

- `queued`：任务已创建，待执行
- `running`：任务执行中
- `completed`：任务执行完成
- `failed`：任务执行失败

### 5.5 状态流转约束

- 未审批通过的需求不能上传数据
- 未上传原始数据的需求不能创建处理任务
- 已归档目录不能发起新需求
- 任务状态只能按定义的合法路径推进
- 非法状态跳转必须显式报错并记录日志

## 8. 与现有前端骨架的衔接

## 9. 后端架构设计

### 9.1 总体方案

一期采用单体全栈架构：

- 前端：Vue 3 + TypeScript + Vite + Pinia + Element Plus
- 后端：FastAPI
- 数据库：PostgreSQL
- ORM：SQLAlchemy
- 迁移：Alembic
- 反向代理：Nginx
- 部署：Docker Compose

### 9.2 后端模块划分

- `auth`：注册、登录、鉴权
- `users`：用户管理、角色分配、状态管理
- `catalogs`：目录管理
- `demands`：需求发起、审批、状态推进
- `assets`：文件上传、下载、元数据登记
- `tasks`：任务创建、状态管理、产物登记

### 9.3 架构原则

- 保持单体部署，避免一期过度复杂化
- 模块边界清晰，便于后续扩展
- 前后端都不做静默兜底，失败要显式暴露

## 10. 文件存储设计

一期采用“数据库存元数据 + 服务器磁盘存文件”的方案。

目录建议：

- `uploads/raw/<demand-id>/`：原始数据文件
- `uploads/processed/<task-id>/`：处理中间产物
- `uploads/delivery/<demand-id>/`：最终交付结果

说明：

- 文件访问必须经过权限校验
- 仅有权限的用户才能下载对应文件
- 文件上传失败时必须返回明确错误

## 11. 指令生成能力的一期接入方式

“指令生成”是一期首个接入的数据处理能力，但采用半接入方式。

### 11.1 一期真实实现的内容

- 在任务中心创建指令生成任务
- 记录输入文件、模型参数、提示词模板、批次大小等配置
- 展示任务状态、日志、结果附件
- 允许通过受控接口推进任务状态

### 11.2 一期不实现的内容

- 不直接调用真实 LLM
- 不做异步任务调度引擎
- 不做自动运行和自动回写结果

### 11.3 一期状态推进方式

- `queued -> running`
- `running -> completed`
- `running -> failed`

要求：

- 每次状态推进都要记录操作来源
- `failed` 必须记录失败原因
- `completed` 时允许登记结果文件或结果说明

## 12. 权限控制设计

权限校验必须由后端主导，前端仅做界面层约束。

### 12.1 前端职责

- 根据用户角色展示可见菜单和操作按钮
- 对无权限页面做路由拦截

### 12.2 后端职责

- 校验登录身份
- 校验角色是否拥有目标操作权限
- 校验用户是否有权访问对应目录、需求、任务和文件

### 12.3 权限原则

- 不信任前端传入的角色信息
- 所有涉及写操作和文件访问的接口都必须做后端权限校验

## 13. 外网访问下的安全基线

部署在实验室服务器且允许外网访问，一期必须满足以下安全要求：

- 全站使用 HTTPS
- 密码仅以摘要形式保存
- 登录鉴权使用令牌机制
- 上传和下载接口都进行权限校验
- 关键操作必须写审计日志
- 不在源代码中硬编码密钥和凭证

## 14. 异常处理策略

遵循“显式失败，不做静默兜底”原则。

### 14.1 登录与审核异常

- 待审核用户登录时提示账号待审核
- 被拒绝用户登录时提示审核未通过
- 被停用用户登录时提示账号已停用

### 14.2 业务状态异常

- 非法状态跳转直接报错
- 不为异常流程构造假成功返回

### 14.3 文件异常

- 文件过大、类型不支持、落盘失败时直接报错
- 元数据入库失败时不伪造上传成功

### 14.4 任务异常

- 半接入任务失败时明确记录失败原因
- 不允许未真实完成的任务直接伪装为成功

## 15. 测试与验收标准

### 15.1 后端单元测试

覆盖内容：

- 注册、审核、登录
- 角色权限校验
- 目录状态流转
- 需求状态流转
- 任务状态流转
- 文件元数据登记

约束：

- 后端单元测试总时长控制在 60 秒内

### 15.2 接口级验证

覆盖流程：

1. 用户注册
2. 管理员审核并分配角色
3. 用户登录
4. 提供者发布目录
5. 汇聚者发起需求
6. 提供者审批并上传数据
7. 汇聚者创建处理任务
8. 汇聚者推进任务并完成交付
9. 使用者下载结果

### 15.3 前端验证

- 路由鉴权正确
- 角色菜单正确
- 页面操作按钮与角色一致
- 表单校验正确
- 上传、下载、状态展示可正常交互

### 15.4 一期验收标准

- 四类角色均可按规则使用系统
- 注册审核与真实登录链路可用
- 数据目录、需求、审批、上传、任务、交付全链路可跑通
- 指令生成任务可创建、可记录配置、可管理状态、可登记结果
- 关键操作均有日志
- 系统可在实验室服务器上部署并通过外网访问

## 16. 二期扩展方向

本节仅作为架构约束说明，不属于一期交付范围。

- 接入真实指令生成执行器
- 接入拆书能力
- 接入修改病句能力
- 引入对象存储
- 引入异步任务执行和调度

一期必须保证这些扩展可以在现有 `tasks` 模块中增量接入，而不需要重做用户体系和数据流转主链路。
