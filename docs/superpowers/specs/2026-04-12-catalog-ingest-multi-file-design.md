# 目录建档即上传多文件设计

## 1. 背景

一期现有实现把“目录”与“原始文件上传”拆成了两段：

1. 数据提供者先创建目录元数据。
2. 数据汇聚者发起需求并获批后，数据提供者再按需求上传文件。

这和当前确认后的业务口径不一致。新的真实业务要求是：

1. 数据提供者在新建目录时同时上传数据集。
2. 一个目录一次可上传多个文件。
3. 数据汇聚者申请的是整个目录的使用权，而不是某个单文件。
4. 目录获批后，数据汇聚者再从目录下勾选若干具体文件创建处理任务。
5. 目录创建后，数据提供者仍可继续追加或删除目录文件。

本设计只重构一期主链路，不扩展到拆书、病句修改或真实异步调度。

## 2. 目标与非目标

### 2.1 目标

- 把原始文件的主归属从 `Demand` 调整为 `Catalog`
- 支持目录创建时多文件上传
- 支持目录后续追加文件与删除文件
- 保持需求仍以 `catalog_id` 为主键进行申请和审批
- 让处理任务支持从同一目录中选择多个输入文件
- 保持当前“文件落盘 + 数据库存元数据”的实现方式，只改变业务入口和关系模型

### 2.2 非目标

- 不把二进制文件改存为数据库 BLOB
- 不实现对象存储、分片上传、断点续传
- 不实现真实 LLM 调度
- 不实现文件级审批，审批粒度仍然是目录

## 3. 新业务流程

### 3.1 数据提供者

1. 在“新建目录”弹窗填写目录元数据。
2. 同时选择一个或多个原始文件上传。
3. 系统创建目录，并为每个文件创建目录文件记录。
4. 数据提供者可在目录详情中继续追加文件或删除尚未被任务使用的文件。
5. 数据提供者发布目录后，目录对汇聚者可见。

### 3.2 数据汇聚者

1. 浏览已发布目录。
2. 针对目录发起需求申请。
3. 数据提供者审批后，该需求直接进入“可处理”状态。
4. 在处理页选择一个已获批需求。
5. 系统展示该需求对应目录下当前可用的目录文件。
6. 汇聚者勾选若干文件，创建处理任务。

### 3.3 数据使用者

- 与一期现状保持一致，只消费最终交付产物，不接触原始目录文件。

## 4. 数据模型调整

### 4.1 Catalog

保留现有目录元数据字段，并新增：

- `asset_count`：只作为接口返回字段，不单独持久化

目录本体仍表示“数据集合”的业务单元，不再承载“上传方式”之外的文件信息。

### 4.2 CatalogAsset

新增目录文件实体，建议命名为 `catalog_assets`。

字段：

- `id`
- `catalog_id`
- `uploaded_by`
- `file_name`
- `file_path`
- `file_size`
- `file_type`
- `uploaded_at`

约束：

- 一个目录至少有一个文件才能创建成功
- 目录发布时必须至少存在一个文件
- 文件真实内容写入 `uploads/catalogs/{catalog_id}/`
- 数据库只保存元数据与相对路径

### 4.3 Demand

`Demand` 仍然按目录申请，不新增文件级字段。

状态机调整为：

- `pending_approval`
- `approved`
- `rejected`
- `data_uploaded`
- `processing`
- `delivered`

其中：

- 目录审批通过后，如果目录中已有文件，需求直接切换到 `data_uploaded`
- 这样处理页无需再等待“按需求上传文件”

保留 `approved` 状态是为了兼容显式状态语义，但在当前主链路中通常会直接进入 `data_uploaded`。

### 4.4 ProcessingTask

任务不再直接持有单个 `input_asset_id`，改为：

- `ProcessingTask` 保留任务自身字段
- 新增 `task_input_assets` 关联表，维护任务与目录文件的多对多关系

`task_input_assets` 字段：

- `id`
- `task_id`
- `catalog_asset_id`

约束：

- 创建任务时至少选择一个目录文件
- 所有输入文件必须属于该需求对应的同一目录
- 只能选择当前存在的目录文件

## 5. 接口设计

## 5.1 目录接口

### `POST /api/catalogs`

改为 `multipart/form-data`。

表单字段：

- `name`
- `data_type`
- `granularity`
- `version`
- `fields_description`
- `scale_description`
- `upload_method`
- `sensitivity_level`
- `description`
- `files`，可重复传入多个文件

行为：

- 校验 `files` 至少一个
- 创建目录
- 写入所有目录文件元数据
- 返回目录详情及 `asset_count`

### `GET /api/catalogs`

返回目录列表时增加：

- `asset_count`

### `GET /api/catalogs/mine`

同样返回 `asset_count`。

### `GET /api/catalogs/{catalog_id}/assets`

用于查看目录下文件。

权限：

- 提供者可查看自己目录的全部文件
- 汇聚者只有在对该目录存在已批准或后续状态需求时才可查看

### `POST /api/catalogs/{catalog_id}/assets`

用于目录追加文件，`multipart/form-data`，支持一次上传多个文件。

权限：

- 仅目录所属提供者

### `DELETE /api/catalogs/{catalog_id}/assets/{asset_id}`

用于删除目录文件。

权限：

- 仅目录所属提供者

限制：

- 若文件已被某处理任务引用，则显式报错，不允许删除
- 删除成功时同步删除磁盘文件

## 5.2 需求接口

### `POST /api/demands`

保持按 `catalog_id` 申请，不增加文件选择。

### `POST /api/demands/{demand_id}/approve`

审批通过后：

- 若目录存在至少一个文件，则状态直接更新为 `data_uploaded`
- 若未来出现空目录异常，则显式返回错误，不做静默降级

### 删除旧需求文件上传主链路

以下接口不再作为一期主链路依赖：

- `GET /api/demands/{demand_id}/assets`
- `POST /api/demands/{demand_id}/assets`

实现上可直接移除，或保留但不再被前端调用；一期最终代码以“前端与服务层不再依赖它们”为准。

## 5.3 处理任务接口

### `POST /api/tasks`

请求体从：

```json
{
  "demand_id": 1,
  "input_asset_id": 3,
  "task_type": "instruction",
  "config": {}
}
```

改为：

```json
{
  "demand_id": 1,
  "input_asset_ids": [3, 5, 8],
  "task_type": "instruction",
  "config": {}
}
```

响应体返回：

- `input_asset_ids`
- 可选返回 `input_assets` 简要信息，便于前端展示

校验：

- 需求必须归当前汇聚者所有
- 需求状态必须是 `data_uploaded` 或 `processing`
- 所选文件必须全部属于该需求对应目录

### `GET /api/tasks`

任务列表中返回所选输入文件 ID 列表，前端可据此展示“已选 N 个文件”。

## 6. 前端页面调整

### 6.1 目录页

目录页新增或调整以下能力：

- 新建目录弹窗支持多文件选择
- 目录列表显示文件数
- 提供者可打开“目录文件管理”弹窗
- 文件管理弹窗支持：
  - 查看目录下文件
  - 追加上传多个文件
  - 删除未被任务使用的文件

### 6.2 需求页

需求页继续按目录发起申请，不增加文件级选择。

展示增强：

- 明确显示“申请的是整个目录”
- 可显示目录文件数，帮助汇聚者判断目录规模

### 6.3 处理页

处理页从“按需求查看上传资产”改为“按需求加载目录文件”：

1. 选择需求
2. 加载该需求对应目录的文件列表
3. 使用多选表格或多选框勾选多个文件
4. 创建任务时提交 `inputAssetIds`

任务列表展示：

- 显示输入文件数量
- 可选展示部分文件名摘要

## 7. 存储与日志

### 7.1 文件存储

- 目录文件统一保存到 `uploads/catalogs/{catalog_id}/`
- 继续沿用当前本地磁盘存储方式
- 保存时生成带随机前缀的文件名，避免覆盖

### 7.2 审计日志

新增或保留以下日志事件：

- `catalog.created`
- `catalog.published`
- `catalog.archived`
- `catalog.asset_uploaded`
- `catalog.asset_deleted`
- `demand.approved`
- `task.created`
- `task.status_updated`
- `task.artifact_registered`

## 8. 错误处理

- 新建目录未上传文件时，返回 422 或 400，显式提示至少上传一个文件
- 追加上传空文件集时，显式报错
- 删除已被任务引用的目录文件时，返回 409
- 汇聚者尝试访问未获批目录文件时，返回 403
- 任务创建未选择文件时，返回 422
- 任务创建时选择跨目录文件时，返回 409

## 9. 测试要求

后端至少覆盖：

- 新建目录时多文件上传成功
- 发布目录前缺少文件失败
- 目录文件追加与删除
- 已获批汇聚者可读取目录文件，未获批不可读取
- 使用多个目录文件成功创建任务
- 删除已被任务引用的文件被拒绝

前端至少覆盖：

- 目录创建 API 正确提交 `FormData`
- 目录页能展示文件数
- 处理页能基于需求加载目录文件并提交多选文件 ID

## 10. 实施边界

本次改动以“一期真实主链路切换成功”为完成标准：

- 目录创建即上传多文件
- 需求审批后无需再做需求级上传
- 处理页可对目录文件做多选输入
- 现有交付下载链路继续可用
